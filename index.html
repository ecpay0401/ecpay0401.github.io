綠界-金流-為什麼Key完卡號送出以後一片空白

發表於 2021-03-24 分類於 金流

為什麼Key完卡號送出以後一片空白?

1.在ECPAY輸入完卡號送出後會將頁面導向各銀行進行3D驗證，

如您使用的瀏覽裝置版本不符合銀行規範(請自行參考各銀行規範)，將可能未知出現錯誤

建議更新瀏覽器版本或更換裝置重新下單購買。

2.若是使用webview方式與Ecpay串接，請注意以下事項

UserAgent 設定:若有指定版本可能會造成在銀行端進行3D驗證時出現以下畫面

因安全性考量如必須要指定版本，建議定期更新，避免無法進行3D驗證而損失訂單。

綠界-模組woocommerce交易失敗！商店尚未開啟收款服務

綠界-模組woocommerce選擇電子地圖遇到『找不到加密金鑰，請確認是否有申請開通此物流方式!』

發表於 2020-09-07 分類於 模組

使用Woocommerce串接綠界服務，選擇門市時遇到『找不到加密金鑰，請確認是否有申請開通此物流方式!』原因如下三點：

1.測試模式若勾選，必須使用測試專用的串接資訊，才能順利選擇門市：

B2C物流類型帳號

特店編號=2000132

HashKey=5294y06JbISpM5x9

HashIV=v77hoKGq4kWxNNIS

C2C物流類型帳號

特店編號=2000933

HashKey=XBERn1YOvpM9nfZc

HashIV=h1ONHk4P4yqbl5LK

反之，若未勾選，則必須使用正式申請的串接資訊，才能選擇門市：

請至廠商管理後台https://vendor.ecpay.com.tw/＞系統開發管理＞系統介接設定 就可取得

2.若需使用物流服務，尚需另外申請，請至https://www.ecpay.com.tw/IntroTransport/Index完成申請流程才能使用

3.待申請完成，請記得當初申請的物流類型是C2C 或 B2C，並設定於上圖的『物流類型』，設定與您申請的相符，才能使用物流服務。

綠界-電子發票 商品明細資料格式不正確

發表於 2020-09-07 分類於 電子發票

商品明細資料格式不正確

此錯誤訊息顯示商品明細『項次』對照有誤，請參考下方，每個參數都需有相同的項目數，若多/少一個就會遇到此錯誤。

ItemCount=1|2|1|1

ItemPrice=20|30|40|-20

ItemAmount=20|60|40|-20

ItemName=商品A|商品B|運費|折扣

ItemWord=個|個|筆|筆

綠界-電子發票 驗證發票金額發現錯誤，與商品合計金額不符的可能原因

發表於 2020-09-07 分類於 電子發票

驗證發票金額發現錯誤，與商品合計金額不符的可能原因

※請留意ItemAmount需統一為含稅金額，所以商品金額需符合以下規則。

1.當vat=1 ItemPrice(含稅)*ItemCount=ItemAmount(含稅) 各項總合並四捨五入=salesAmount(含稅)

EX：

ItemPrice=10|-20|30

ItemCount=2|1|2

ItemAmount=20|-20|60

salesAmount=60

2.當vat=0 ItemPrice(不含稅)ItemCount1.05=ItemAmount(含稅) 各項總合並四捨五入=salesAmount(含稅)

EX：

ItemPrice=2980|-566|1

ItemCount=2|1|1

ItemAmount=6258|-594.3|1.05

salesAmount=5665

綠界-金流 常見付款狀態未更新原因

發表於 2020-09-07 分類於 金流

常見付款狀態未更新原因
若消費者已經付款，綠界廠商管理後台也已更新付款

但您的系統後台(Woocommerec、Opencart、Shopline、QDM、美安….平台)

付款狀態未被更新，請程式人員檢查以下設定:

(1)接收付款通知的伺服器是否異常

● 如您正在舉辦搶購活動，請務必確保伺服器可短時間大量接收綠界回傳的付款結果通知

Ex:AWS、台灣固網….等其他網路業者，預設防火牆機制，短期間接收大量回傳會視為攻擊事件，請將綠界IP加入白名單

(2)近期是否有變更接收付款通知的ReturnURL網址

(3)接收付款結果的ReturnURL程式是否有判斷交易成功(RtnCode=1表示付款成功，其餘代碼為失敗)

(4)貴司系統收到綠界回傳結果，是否有正確回應1|OK

(5)因資安政策規定綠界僅提供443 PORT且需使用合法的DomainName進行設定

上述原因確認完畢，仍然無法排除此問題請提供以下資訊

並Email至techsupport@ecpay.com.tw

(1)商店代號(MerchantID)

(2)廠商交易編號(MerchantTradeNo)

(3)交易日期(MerchantTradeDate)

綠界-金流 綠界帳戶餘額低於退刷金額或為負數導致無法退刷及建立物流訂單

發表於 2020-09-07 更新於 2020-09-14 分類於 金流

若您的帳戶餘額低於退刷金額為負數

將無法進行退刷以及建立物流訂單

請至綠界廠商管理後台進行預付動作，

如需進行退刷作業，服務項目請選擇”信用卡退刷款” 請存入足夠退刷的金額

如需進行物流費用預存，服務項目請選擇”預付物流款”請存入足夠每周二結算物流費用使用金額。

綠界-物流 電子地圖無法顯示

發表於 2020-09-05 更新於 2020-11-26 分類於 物流

此政策下，所有cookies將需要清楚標示Samesite值,

若未標示,Chrome會強制將Samesite指定為LAX

若您的網站未指定Samesite=None將可能出現無法預期的錯誤。

Samesite=Strict Samesite=LAX Samesite=None (允許跨網站存取)

例如:結帳完被登出、物流電子地圖無法將資料回寫

如何自行檢測?
開啟您的網站>Ctrl+Shift+I

如出現下圖表示未設置，預設為(LAX)

（請注意，在Chrome 85之前，每次加載頁面時，DevTools中都會向JavaScript控制台發送有關不兼容Cookie的消息。在Chrome 85中，這些消息已從控制台中刪除。)


自家網站的調整方式
1.使用HTTPS 加強交易雙方安全

2.將Set-Cookie 的行為都加上 SameSite 的屬性

SameSite=None + HTTPS：允許跨網域存取，但是若沒 HTTPS 則預設「拒絕」跨網域存取
無效：Set-Cookie: remote_session=abc123; SameSite=None
有效：Set-Cookie: remote_session=abc123; SameSite=None; Secure

PHP 7.2 以前的 SameStie Cookie 寫法
header(‘Set-Cookie: cross-site-cookie=name; SameSite=None; Secure’);

header(‘Set-Cookie: cookie2=name; SameSite=None; Secure’, false);

PHP 7.3 以後的 SameStie Cookie 寫法
setcookie(‘cross-site-cookie’, ‘name’, [‘samesite’ => ‘None’, ‘secure’ => true]);

setcookie（’cookie2’，’name’，[‘samesite’=>’None’，’secure’=> true]）;

設定 .htaccess
Header always edit Set-Cookie ^(.)$ “$1;HttpOnly;Secure;SameSite=None”
設定 Apache httpd.conf
Header always edit Set-Cookie ^(.)$ “$1;HttpOnly;Secure;SameSite=None”

設定完成，請記得重啟服務及清除瀏覽記錄。

網站還沒調整好，消費者端暫時的解決方式
Step 1. 於 Chrome 瀏覽器網址列輸入 chrome://flags
Step 2. 搜尋Samesite
Step 3. 將以下三個設定設定為Disabled
Step 4. 按下Relaunch,重啟瀏覽器

Safari

關閉 防止跨網站的追蹤功能>重啟瀏覽器
